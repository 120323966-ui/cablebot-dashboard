# 前端系统修改清单

本文档基于当前前端代码与论文中“电缆排管巡检机器人远程操作终端”的理想构造整理。优先级按影响范围排序：先保证项目可构建，再修跨页面状态一致性，最后优化交互、可视化和工程化边界。

## P0：先修复构建与类型错误

### 1. 修正 Command 页任务状态类型不一致

- 现象：`npm run build` 失败，`CommandMission.status` 类型只允许 `running | paused | attention`，但 mock 数据会返回 `queued`。
- 涉及文件：
  - `src/types/command.ts`
  - `src/mocks/data/commandCenter.ts`
- 建议修改：
  - 若“待命中机器人”是合法任务状态，则把 `MissionStatus` 扩展为 `running | paused | attention | queued`。
  - 同步检查 Command 页所有状态渲染和控制按钮逻辑，避免 `queued` 被当成 `running`。
- 验收标准：
  - `npm run build` 不再因 `queued` 报错。
  - R3/R4 这类待命机器人进入实时巡检页时，界面明确显示“待命/未开始”，不显示为正在巡检。

### 2. 修正网络质量枚举不一致

- 现象：mock 返回 `fair | poor`，但 `NetworkQuality` 只允许 `excellent | good | unstable`。
- 涉及文件：
  - `src/types/command.ts`
  - `src/mocks/data/commandCenter.ts`
  - `src/pages/Command/RightCommandRail.tsx`
- 建议修改：
  - 方案 A：把 mock 的 `fair/poor` 映射成 `good/unstable`。
  - 方案 B：扩展类型并补齐 `networkLabel`、`networkTone`。
  - 推荐方案 A，保持语义简单。
- 验收标准：
  - `npm run build` 不再因 `networkQuality` 报错。
  - 网络质量显示与颜色语义一致。

### 3. 清理 TypeScript 未使用变量和错误调用

- 现象：多个文件存在未使用 import、未使用变量、函数参数类型传错等问题。
- 涉及文件：
  - `src/mocks/data/dashboardHome.ts`
  - `src/mocks/data/alerts.ts`
  - `src/mocks/data/reports.ts`
  - `src/mocks/data/sharedSeed.ts`
  - `src/pages/Alerts/AlertDetail.tsx`
  - `src/pages/Command/RightCommandRail.tsx`
  - `src/pages/Command/VoiceOverlay.tsx`
  - `src/pages/History/HistoryPage.tsx`
  - `src/pages/Spatial/PointCloudView.tsx`
- 建议修改：
  - 删除未使用 import 和变量。
  - `RightCommandRail.tsx` 中 `trendArrow(sensor)` 应改为 `trendArrow(sensor.trend)`。
  - `VoiceOverlay.tsx` 的 history 状态需要显式保留字面量类型，例如 `status: 'executed' as const`。
- 验收标准：
  - `npm run build` 通过。
  - `npm run lint` 的未使用变量错误清零。

### 4. 修复 React Hook / Fast Refresh 规则问题

- 现象：lint 报出 refs 在 render 期间写入、Fast Refresh 文件导出混杂等问题。
- 涉及文件：
  - `src/hooks/useRegisterVoiceKeys.ts`
  - `src/context/DashboardContext.tsx`
  - `src/hooks/useRealtimeDashboard.ts`
  - `src/hooks/useRealtimeCommandCenter.ts`
  - `src/hooks/useSpatial.ts`
- 建议修改：
  - `useRegisterVoiceKeys.ts` 中 `statusRef.current = voice.state.status` 移入 `useEffect`。
  - 将 `useDashboardContext` 拆到独立文件，或调整 ESLint 规则/导出结构，避免 `react-refresh/only-export-components` 报错。
  - 将 `Boolean(data)` 这类复杂依赖提取为变量。
- 验收标准：
  - `npm run lint` 通过或仅剩可接受的 MSW worker 警告。

## P1：统一运行态，解决跨页面状态不同步

### 5. Spatial 页接入全局告警状态

- 现象：告警页确认/关闭告警后，首页和 Command 页会同步，但 Spatial 页仍使用 `/api/dashboard/spatial` 的独立快照，拓扑图上的告警点不会同步变化。
- 涉及文件：
  - `src/hooks/useSpatial.ts`
  - `src/pages/Spatial/SpatialPage.tsx`
  - `src/mocks/data/spatial.ts`
  - `src/context/DashboardContext.tsx`
- 建议修改：
  - Spatial 的 `alerts` 从 `DashboardContext.alerts` 派生，而不是只使用接口初始快照。
  - 增加 `AlertItem -> PipeAlert` 的转换函数，复用 `mapAlertType`。
  - 关闭告警后，拓扑图应立即隐藏对应告警点，传播链也应重新计算或清空。
- 验收标准：
  - 在告警页关闭 `AL-301` 后，空间定位页不再显示该告警点。
  - 从告警页进入空间定位页时，URL `?alert=AL-xxx` 能定位到最新状态。

### 6. 把急停从“暂停”升级为全局状态

- 现象：Command 页本地使用 `attention` 表示急停/注意状态，但写回全局 `activeTask.status` 时变成 `paused`，首页和快捷键无法区分暂停与急停。
- 涉及文件：
  - `src/types/dashboard.ts`
  - `src/pages/Command/CommandPage.tsx`
  - `src/hooks/useKeyboardShortcuts.ts`
  - `src/pages/HomeOverview/MissionStrip.tsx`
  - `src/pages/HomeOverview/StatusBar.tsx`
- 建议修改：
  - 将 `ActiveTask.status` 扩展为 `running | paused | queued | completed | emergency`，或统一采用 `attention`。
  - `Escape` 快捷键和急停按钮写入全局急停状态。
  - 急停状态下，普通控制按钮置灰，界面显示执行时间和恢复条件。
- 验收标准：
  - Command 页触发急停后，首页、顶部状态、事件流都能看到急停状态。
  - 暂停和急停在颜色、文案和恢复操作上有明确区别。

### 7. 修正 Command 事件流的告警状态映射

- 现象：`closed` 告警被映射为 Command 页事件状态 `processing`，语义错误。
- 涉及文件：
  - `src/types/command.ts`
  - `src/pages/Command/CommandPage.tsx`
  - `src/pages/Command/RightCommandRail.tsx`
- 建议修改：
  - 将 `EventStatus` 扩展为 `new | acknowledged | closed`，或至少不要把 `closed` 映射为 `processing`。
  - Command 页事件流显示“已关闭/已确认/待处置”，与告警页一致。
- 验收标准：
  - 在告警页关闭一条告警后，Command 页事件流同步显示“已关闭”。

### 8. 首页 KPI 改为从全局告警实时派生

- 现象：首页 `KpiStrip` 使用初始 `data.kpis`，实时新增告警或确认/关闭告警后，KPI 可能与顶部铃铛和告警页不一致。
- 涉及文件：
  - `src/context/DashboardContext.tsx`
  - `src/pages/HomeOverview/HomeOverviewPage.tsx`
  - `src/pages/HomeOverview/KpiStrip.tsx`
- 建议修改：
  - 在 Context 中派生实时 KPI，或在首页根据 `alerts`、`robots`、`activeTask` 计算 KPI。
  - 保证“待处置告警”“高风险区段”等关键指标与全局告警状态一致。
- 验收标准：
  - 新增告警后：首页 KPI、顶部铃铛、告警页数量一致。
  - 关闭告警后：首页待处置数量立即下降。

## P1：完善告警处置闭环

### 9. 告警详情使用归并后的最新证据

- 现象：列表已经显示 `latestEvidence/latestOccurredAt/repeatCount`，但详情页仍显示原始 `evidence/occurredAt`。
- 涉及文件：
  - `src/pages/Alerts/AlertDetail.tsx`
  - `src/pages/Alerts/AlertList.tsx`
- 建议修改：
  - 详情页统一使用 `displayEvidence = latestEvidence ?? evidence`。
  - 显示累计次数、首次发生时间、最新发生时间。
- 验收标准：
  - 重复告警被归并后，详情页能看到最新值和累计次数。

### 10. 处置备注需要按告警 ID 存储并进入全局记录

- 现象：备注是 `AlertDetail` 内部本地状态，切换告警后容易混淆，也不会进入历史和报告。
- 涉及文件：
  - `src/pages/Alerts/AlertDetail.tsx`
  - `src/context/DashboardContext.tsx`
  - `src/types/alerts.ts`
  - `src/pages/Reports/ReportsPage.tsx`
- 建议修改：
  - 在 Context 中增加 `alertNotes: Record<alertId, AlertNote[]>`。
  - `AlertDetail` 根据当前 `alert.id` 读取/写入备注。
  - 报告生成页读取处置备注。
- 验收标准：
  - 切换不同告警时备注不会串号。
  - 报告预览包含对应告警的处置备注。

### 11. 关键操作补确认弹窗

- 现象：告警“直接关闭”、急停、人工接管、策略执行等动作风险较高，但部分操作没有二次确认。
- 涉及文件：
  - `src/pages/Alerts/AlertDetail.tsx`
  - `src/pages/Command/BottomControlDock.tsx`
  - `src/components/ui/ConfirmDialog.tsx`
- 建议修改：
  - 关闭 critical 告警、急停恢复、人工接管、策略执行使用 `ConfirmDialog`。
  - 低风险操作保持直接执行，避免确认链路过长。
- 验收标准：
  - critical 告警不能一键误关闭。
  - 急停和人工接管都有清晰确认/反馈路径。

## P2：对齐论文中的交互与可视化细节

### 12. 实时巡检页补“通信异常/延迟”受限状态

- 现象：实时页有视频延迟和网络质量展示，但没有完整模拟“通信中断时最后一帧、控制置灰、本地功能保留”的状态。
- 涉及文件：
  - `src/pages/Command/CenterVideoStage.tsx`
  - `src/pages/Command/BottomControlDock.tsx`
  - `src/hooks/useRealtimeCommandCenter.ts`
- 建议修改：
  - 引入 `networkState: online | degraded | offline`。
  - offline 时视频区显示最后时间戳，控制按钮置灰，仅保留截图/查看详情/历史记录等本地功能。
- 验收标准：
  - 能通过 mock 触发通信异常，并看到视频、状态卡、控制区同步变化。

### 13. 行进策略建议需要和拓扑风险联动

- 现象：论文中“继续/减速/停止”应由告警等级、机器人位置、前方未处置风险共同约束；当前更偏静态按钮。
- 涉及文件：
  - `src/pages\Command\BottomControlDock.tsx`
  - `src/pages\Spatial\SpatialInfoPanel.tsx`
  - `src/utils/propagation.ts`
  - `src/context/DashboardContext.tsx`
- 建议修改：
  - 增加策略对象：`continue | slow | stop | takeover`、依据、来源告警、是否已确认。
  - 当机器人位于或驶向未处置 critical 告警区段时，策略卡显示“建议停止/接管”。
- 验收标准：
  - 选择 B3 critical 告警后，实时巡检页和空间定位页能看到一致的策略依据。

### 14. 空间传播链加入“规则参数说明”

- 现象：传播链已有算法和视觉表达，但界面没有说明时间窗口、跳数和方向规则，操作员难以判断这是事实关联还是推测范围。
- 涉及文件：
  - `src/pages/Spatial/SpatialInfoPanel.tsx`
  - `src/utils/propagation.ts`
- 建议修改：
  - 在侧栏显示：异常类型、传播方向、时间窗口、最大关联跳数。
  - 明确区分“已发生同类告警”和“推测影响范围”。
- 验收标准：
  - 传播链视图中能看出依据，而不是只看到高亮效果。

### 15. 历史分析与实时告警语义分离

- 现象：历史热力图容易被误读为当前危险等级。
- 涉及文件：
  - `src/pages/History/HistoryPage.tsx`
  - `src/pages/HomeOverview/HeatmapChart.tsx`
- 建议修改：
  - 历史页热力图明确标注“历史风险集中度，不代表当前实时告警等级”。
  - 首页热力图与实时告警颜色不要完全混用。
- 验收标准：
  - 操作员能区分“历史高风险区段”和“当前未处置告警”。

## P2：报告与历史复盘完善

### 16. 报告生成接入当前告警处置记录

- 现象：报告页主要使用 reports mock 数据，未直接读取当前运行态中的最新确认、关闭、备注、急停等事件。
- 涉及文件：
  - `src/hooks/useReports.ts`
  - `src/pages/Reports/ReportsPage.tsx`
  - `src/context/DashboardContext.tsx`
- 建议修改：
  - 报告预览合并全局告警状态和处置备注。
  - 增加“本次任务事件流”：告警确认、关闭、人工接管、急停、通信中断。
- 验收标准：
  - 当前刚确认/关闭的告警能出现在报告预览中。

### 17. 历史分析页增加事件联动入口

- 现象：历史页有图表和表格，但与空间定位、告警详情的联动还可以更强。
- 涉及文件：
  - `src/pages/History/HistoryPage.tsx`
  - `src/router/routes.tsx`
- 建议修改：
  - 点击历史告警，跳转 `/alerts?id=AL-xxx` 或 `/spatial?segment=B3`。
  - 点击区段热力图，跳转空间页并选中区段。
- 验收标准：
  - 历史复盘可以快速进入告警详情和空间核查。

## P3：工程化边界与可维护性

### 18. 生产预览环境需要 API 兜底

- 现象：`main.tsx` 只在 `DEV` 启动 MSW，`npm run preview` 时 `/api/dashboard/*` 没有后端会请求失败。
- 涉及文件：
  - `src/main.tsx`
  - `src/mocks/browser.ts`
  - `vite.config.ts`
- 建议修改：
  - 增加 `VITE_ENABLE_MOCKS=true` 开关，允许 preview 环境启用 MSW。
  - 或为 preview 配置本地 mock server。
- 验收标准：
  - `npm run build && npm run preview` 后页面可以正常加载 mock 数据。

### 19. README 编码和内容需要修复

- 现象：README 在当前环境输出为乱码，且内容仍描述“只完成首页和占位页”，与当前代码不一致。
- 涉及文件：
  - `README.md`
- 建议修改：
  - 统一保存为 UTF-8。
  - 更新当前页面完成度、运行方式、mock 策略和已知限制。
- 验收标准：
  - PowerShell/编辑器中读取 README 不再乱码。
  - README 反映当前六页面系统，而不是旧脚手架状态。

### 20. 抽出统一的领域转换层

- 现象：`AlertItem`、`PipeAlert`、`CommandEvent` 之间的转换散落在多个页面中，容易出现状态和字段语义不一致。
- 涉及文件：
  - `src/types/dashboard.ts`
  - `src/types/spatial.ts`
  - `src/types/command.ts`
  - 建议新增：`src/utils/domainAdapters.ts`
- 建议修改：
  - 新增转换函数：
    - `alertToPipeAlert`
    - `alertToCommandEvent`
    - `activeAlertToAlertItem`
  - 所有页面复用这些转换函数。
- 验收标准：
  - 状态映射、类型映射、严重等级映射只维护一份。

## 建议执行顺序

1. 修 P0：让 `npm run build` 和 `npm run lint` 通过。
2. 修 P1：统一告警、任务、急停、空间定位状态。
3. 修告警闭环：最新证据、备注、确认弹窗、状态回写。
4. 修可视化细节：通信异常、策略建议、传播链依据、历史/实时语义分离。
5. 修工程化边界：preview mock、README、领域转换层。
